## Critical Issues (Fix Now)

### 1. NEW: Admin Approve Endpoint — Auth Check Always Fails (Unauthenticated Access)
**`app/api/admin/approve/route.ts:17`** — The check is `if (!session?.isAdmin)`, but `getSession()` queries the DB and returns the raw user row which has the field `is_admin` (snake_case). The property `isAdmin` is always `undefined`, so this guard never blocks an unauthenticated caller. Any user — logged in or not — can call `POST /api/admin/approve` and approve any contact request, creating a real user account and triggering a password email.

**Fix:** Change `session?.isAdmin` → `session?.is_admin` on line 17 to match the DB column name.

---

### 2. Forgeable Session Token (Authentication Bypass) — UNRESOLVED from 2026-03-27
**`lib/auth.ts:21-22`** and **`middleware.ts:15-20`** — The session token is `base64(userId:email:isAdmin)`. Any attacker can encode `1:admin@apexprotocol.co.za:1` in base64 and set it as the `session_token` cookie to gain admin access. There is no HMAC or signature of any kind.

**Fix:** Sign tokens using `crypto.createHmac('sha256', process.env.AUTH_SECRET).update(payload).digest('hex')` and verify the signature on every request in both `lib/auth.ts` (for API routes) and `middleware.ts` (for page routing).

**Fix:** Replace with `process.env.SMTP_USER` / `process.env.SMTP_PASS`. Rotate the password immediately — it is in git history permanently unless history is rewritten.

### 5. Command Injection in Contact Form — UNRESOLVED from 2026-03-27
**`app/api/contact/route.ts:53-56`** — User-supplied `firstName` and `lastName` are interpolated directly into a shell command passed to `execAsync()`. A user can set `firstName` to `"; rm -rf /; echo "` to execute arbitrary commands on the server.

**Fix:** Remove the `himalaya` CLI invocation entirely. Use the existing `nodemailer` transporter from `lib/email.ts` to send the contact notification email. This also fixes the silent production failure (himalaya is not installed on Vercel/Cloudflare).

## Warnings (Fix Soon)

**Fix:** Consolidate everything to `lib/db-pg.ts`. Remove `lib/db.ts`.

**91 instances** of `: any`, `as any`, or `<any>` across the codebase (up from "40+" last cycle). With `strict: true` in tsconfig, these are suppressed type errors that can hide bugs. Affected files include `app/admin/orders/page.tsx`, `app/api/admin/sync/route.ts`, `app/api/checkout/route.ts`, and 15+ more.

### 3. Client-Submitted Prices Trusted at Checkout — UNRESOLVED from 2026-03-27

**Fix:** Ignore client-submitted prices. Re-fetch product `sell_price` from DB for each `item.id` and recalculate `subtotal` server-side. Calculate `shipping.cost` using `lib/shipping.ts:calcShipping()`.

### 5. `require()` in TypeScript Files — UNRESOLVED from 2026-03-27
**`app/api/admin/approve/route.ts:42,47`**, **`app/api/contact/route.ts:50,56`**, **`lib/db.ts:53`** — CommonJS `require()` is used inside ES module TypeScript files. This breaks tree-shaking and is inconsistent with the rest of the codebase which uses ESM imports.

**Fix:** Replace `require('bcryptjs')` with `import bcrypt from 'bcryptjs'` at the top of each file. Replace `require('fs')` with `import { writeFileSync, unlinkSync } from 'fs'`.

### 6. `@types/react@19` vs `react@18` Major Version Mismatch
**`package.json:34`** — `@types/react` is pinned to `^19.2.14` but `react` is `^18.3.1`. React 19 type definitions are incompatible with React 18 runtime, which can cause phantom TypeScript errors or hide real ones.

**Fix:** Downgrade `@types/react` to `^18.3.x` to match the installed React version: `npm install -D @types/react@^18.3.0 @types/react-dom@^18.3.0`.

### 7. Raw `<img>` Tags Bypassing Next.js Image Optimization
**`components/ProductCard.tsx:44`**, **`app/cart/page.tsx:41`**, **`app/checkout/page.tsx:312`**, **`app/admin/products/page.tsx:97`** — Four places use raw `<img>` instead of `next/image`. Combined with `images.unoptimized: true` in `next.config.js`, no image optimization, lazy loading, or WebP conversion happens.

**Fix:** Replace with `<Image>` from `next/image`. Remove `unoptimized: true` from `next.config.js` (or keep it only for the Cloudflare Pages deployment target, not Vercel).

### 8. Empty Catch Blocks Silently Swallow Errors
**`app/admin/orders/page.tsx:119`**, **`app/api/contact/route.ts:31`**, **`app/checkout/page.tsx:86`** — Three empty `catch {}` blocks make runtime errors invisible. The contact route's empty catch on the `ALTER TABLE` migration is particularly risky — it silently ignores ALL errors including ones unrelated to the migration.

**Fix:** At minimum, `catch (e) { console.error(..., e); }`. For the migration catch in contact route, check `e.message` for "already exists" before swallowing.

4. **Password change endpoint missing** — No `/api/auth/change-password` route. Users receive auto-generated temp passwords via `approve/route.ts` and have no way to change them. This is a UX blocker for approved members.

5. **Design system tokens mismatch** — Spec colors (`#00E5CC`, `#8B5CF6`, `#0a0f1a`, `#141930`) differ from implementation (`#00d4ff`, `#111827`, `#0a0a0a`). Neither `card-primary`, `card-secondary`, nor `card-tertiary` CSS classes exist.

6. **Rename deprecated Next.js config key** — `next.config.js:3` uses `serverComponentsExternalPackages` which was renamed to `serverExternalPackages` in Next.js 14.1+. The old key still works but will be removed in Next 15.

7. **Age gate not applied globally** — `components/AgeGate.tsx` exists and is rendered in `app/layout.tsx`, but the component itself may not persist across navigation (CSR state). Verify the gate is enforced correctly on all protected pages given this is a controlled-substance platform.

8. **Nodemailer transporter created per-request** — `app/api/auth/register/route.ts:5-10`, `app/api/admin/approve/route.ts:6-11`, and `app/api/admin/members/route.ts:5-10` each create a new SMTP transporter on every request instead of reusing the one in `lib/email.ts`. This creates unnecessary SMTP connection overhead and duplicates the hardcoded credentials.
