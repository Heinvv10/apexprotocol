## Metric Corrections vs Previous Reports

Two persistent inaccuracies corrected this week:

1. **`any` violations**: Previous reports used `grep -rn ": any"` without excluding commented lines. Correct methodology (`grep | grep -v "// "`) yields **178**, not 184. Score unchanged since no code was modified.

2. **`console.log` leaks**: Previous count of 5 was false-positive. All 5 grep hits were string literals inside comments in `src/lib/logger.ts` and `src/lib/utils/retry.examples.ts` — not actual `console.log()` calls. Actual count remains **0**.

3. **`/api/generate`, `/api/jobs`, `/api/gamification` — NOT fully unprotected**: These routes are NOT listed in `publicRoutes` in `src/middleware.ts`. In production (when `SUPABASE_AUTH_CONFIGURED=true`), the middleware's `productionMiddleware` function requires a valid Supabase session for all non-public routes, returning HTTP 401 for unauthenticated requests. The routes still lack **internal** auth checks (defense-in-depth), and they remain fully unprotected in dev mode (`devMiddleware` has no session check), but calling them "unprotected" overstated the production risk.

## Critical Issues (Fix Now)

1. **`@tanstack/react-query` missing from production dependencies — app will crash in prod build** — `package.json` — `@tanstack/react-query` is NOT in `node_modules` and NOT in `dependencies` or `devDependencies` (only `@tanstack/react-query-devtools` is a devDep). **51 files** import it. Any `npm ci --omit=dev` / Docker production build will fail at runtime with `Cannot find module '@tanstack/react-query'`. *Unresolved for 10 consecutive weeks.*
   **Fix:** `npm i @tanstack/react-query`

2. **XSS — unsanitized HTML via `dangerouslySetInnerHTML`** — `src/components/create/content-preview-modal.tsx:83` — `__html: content || "..."` receives raw AI-generated text with zero sanitization. A prompt-injected `<script>` or `<img onerror=...>` executes in the user's browser. *Unresolved for 10 consecutive weeks.*
   **Fix:** `npm i isomorphic-dompurify @types/dompurify` then:
   ```tsx
   dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(content ?? "") }}
   ```

3. **In-memory export job state lost on every restart + no internal auth on history** — `src/app/api/export/route.ts:42` — `const exportJobs = new Map<string, ExportJob>()` is module-level state lost on every pod restart. `src/app/api/export/history/route.ts` has no internal auth check (relies solely on middleware session). *Unresolved for 9 weeks.*
   **Fix:** Move to `@upstash/redis` with TTL hash; add `createSupabaseServerClient().auth.getUser()` guard to handlers.

4. **GraphQL Playground schema introspection exposed without route-level auth** — `src/app/api/graphql/route.ts` — The `yoga` handler is exported directly as `GET` and `POST` with no auth wrapper. GET requests can access the GraphQL Playground UI and run introspection queries to enumerate the full schema (types, queries, mutations, field names) without any authentication. Resolvers call `context.requireAuth()`, but introspection itself may be unauthenticated depending on yoga config. **NEW finding this week.**
   **Fix:** Wrap the handler with Supabase session check, or disable introspection in production:
   ```ts
   const yoga = createYoga({ schema, graphiql: process.env.NODE_ENV !== 'production' })
   ```

5. **`/api/jobs` — BullMQ internals exposed without internal auth** — `src/app/api/jobs/route.ts:1` — 335-line route exposes full queue state, job payloads, cancel/retry management. No auth imports in file; relies entirely on middleware session (no defense-in-depth). In dev mode (no Supabase config) this is fully unprotected. *Unresolved for 9 weeks.*

6. **`/api/gamification` family — XP, achievements, badges, challenges writeable without internal auth** — `src/app/api/gamification/route.ts`, `.../badges/route.ts`, `.../challenges/route.ts` — No auth imports in any of these three files; relies entirely on middleware. In dev mode, any caller can inflate XP or award achievements for any `userId` in the request body. *Unresolved for 9 weeks.*

7. **Stale Clerk env vars active in docker-compose.yml** — `docker-compose.yml:31-33` — `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`, `CLERK_SECRET_KEY`, `CLERK_WEBHOOK_SECRET` remain in the production container env. The app migrated to Supabase in commit `f716413`. These env vars do nothing but add confusion and could trigger a future mis-wired Clerk integration.
   **Fix:** Remove the three Clerk env var entries from `docker-compose.yml`.
